--- Слайд 5. Архитектура системы (1:50 — 2:20) ---
В качестве архитектурного паттерна серверной части выбран «модульный монолит». В отличие от классического монолита, вся бизнес-логика разделена на слабосвязанные модули:
- Auth — аутентификация пользователей;
- Projects — создание и ведение проектов;
- Members — управление ролями участников;
- Tasks — ведение досок, колонок, задач и комментариев;
- Notify — информирование пользователей;
- Admin — панель администрирования.

Каждый модуль отвечает за свою предметную область. Такая изоляция позволяет дорабатывать отдельные модули независимо друг от друга, а при необходимости — легко выделить любой из них в отдельный микросервис.

--- Слайд 6. Структура модуля (2:20 — 2:50) ---
Внутри каждого модуля бэкенда реализована классическая трехслойная архитектура:
1. Слой обработчиков (Handler) принимает входящие HTTP-запросы, выполняет валидацию входных данных и отправляет клиенту структурированный JSON-ответ.
2. Слой сервисов (Service) инкапсулирует бизнес-логику, координирует операции и управляет транзакциями базы данных.
3. Слой репозиториев (Repository) отвечает исключительно за выполнение SQL-запросов к PostgreSQL, полностью скрывая детали хранения данных.

ЧАСТЬ 3. ОТВЕТЫ НА ВОЗМОЖНЫЕ ВОПРОСЫ КОМИССИИ (Q&A)
========================================================================

1. Как обеспечивается безопасность WebSocket-соединения?
Ответ:
WebSocket-подключение защищено шифрованием TLS (протокол WSS). JWT-токен передается в качестве query-параметра (?token=...) при инициализации соединения, поскольку стандартный браузерный API WebSockets не поддерживает передачу кастомных заголовков. 
На сервере в Middleware происходит разбор URL, извлечение токена, его декодирование и проверка подписи секретным ключом. Если токен невалиден или просрочен, сервер отклоняет запрос на соединение (возвращает статус 401).

2. Как реализована ролевая модель доступа в коде бэкенда?
Ответ:
Ролевая модель доступа базируется на числовых рангах (весах) ролей, определенных в модуле members: observer — 0, executor — 1, manager — 2, admin — 3. 
При выполнении запроса сервис запрашивает роль пользователя в проекте из таблицы project_members. В коде написаны предикаты проверки прав (например, CanEditTasks(role)), которые проверяют, соответствует ли ранг пользователя требуемому уровню доступа. Владелец проекта (owner_id) имеет абсолютный доступ и проверяется отдельно.

3. В чем преимущества выбранного стейт-менеджера Zustand по сравнению с Redux?
Ответ:
Zustand — это современный, хуковый стейт-менеджер. В отличие от Redux, он не требует оборачивать компоненты в Provider (что исключает лишние ререндеры дерева React-компонентов), не имеет громоздкого шаблона кода (boilerplate), и позволяет создавать изолированные хранилища данных. Это значительно упростило логику обновления состояния доски при получении асинхронных данных по WebSockets.

4. Каким образом в PostgreSQL обеспечивается целостность данных при удалении проектов?
Ответ:
Для обеспечения целостности данных используются ограничения внешних ключей (FOREIGN KEY) со спецификацией ON DELETE CASCADE. При удалении записи проекта СУБД PostgreSQL автоматически удаляет все связанные с ним доски, колонки, задачи, комментарии и записи участников. Это гарантирует отсутствие в БД неиспользуемых "сиротских" записей без необходимости писать сложный код удаления на бэкенде.

5. Почему в качестве СУБД выбрана PostgreSQL, а не NoSQL (например, MongoDB)?
Ответ:
Данные в системе носят строго структурированный и реляционный характер: сущности имеют четкие связи (проекты -> участники -> задачи). PostgreSQL гарантирует транзакционность (ACID), исключающую частичную запись данных при сбоях, и позволяет выполнять высокопроизводительные объединения данных через JOIN для сбора статистики на дашборде. В NoSQL-системах пришлось бы дублировать данные, что привело бы к рискам рассинхронизации.

6. Как в Go-сервере предотвращается проблема SQL-инъекций?
Ответ:
Все SQL-запросы в репозиториях Go используют подготовленные выражения (Prepared Statements) с плейсхолдерами параметров (например, $1, $2). Драйвер PostgreSQL передает параметры отдельно от текста запроса, что делает невозможным внедрение вредоносного SQL-кода через пользовательский ввод.

7. Каким образом шифруются пароли пользователей?
Ответ:
При регистрации пароль хэшируется криптостойкой функцией bcrypt с фактором стоимости (salt rounds), равным 10. Алгоритм автоматически генерирует уникальную криптографическую соль для каждого пароля. Это предотвращает взлом паролей с использованием готовых радужных таблиц и делает метод полного перебора (brute-force) неэффективным.

8. Как устроена работа с транзакциями в Go?
Ответ:
При выполнении операций, изменяющих несколько строк или таблиц (например, добавление задачи с пересчетом порядка остальных задач в колонке), сервис вызывает db.BeginTx(). Все SQL-запросы выполняются в контексте созданной транзакции. 
При возникновении любой ошибки вызывается tx.Rollback(), отменяющий все изменения. При успешном выполнении всех шагов вызывается tx.Commit().

9. Как приложение защищено от атак типа XSS?
Ответ:
React по умолчанию экранирует любые строковые переменные перед выводом их на веб-страницу, преобразуя потенциально опасные спецсимволы в безопасные HTML-сущности. В коде не используются небезопасные методы прямого рендеринга HTML (такие как dangerouslySetInnerHTML), что исключает внедрение вредоносных скриптов через названия задач или комментарии.

10. Почему вы выбрали Go в качестве языка для бэкенда?
Ответ:
Go компилируется в один легковесный бинарный файл, работает без виртуальной машины и потребляет минимум системных ресурсов. Он обладает высокой производительностью и встроенной поддержкой многопоточности (горутины и каналы), что критично для обслуживания тысяч одновременных WebSocket-соединений без зависаний сервера.
